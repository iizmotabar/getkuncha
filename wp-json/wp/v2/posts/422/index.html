Analytics debugging could be a very panic-inducing process once an issue is discovered. It\u2019s more or less a hair on fire situation to find the issue and a way to fix it. This process can take from a few hours to few days depending on the data you debug and the tools you are using.<\/p>\n

As a consultant figuring out a right method to perform this action is necessary especially since you are often working with client developers and not interacting with the code or limited in what you have access to. That means you only have to focus on doing it the right way so you can save time but also be able to point people to where the error might be.<\/p>\n

To help you figure out this process, I will discuss various approaches and access points you have available to debug in the browser, how you should go about the process and what tools you can use.<\/p>\n

Analytics Debugging, Why?<\/strong><\/h2>\n
    \n
  • Make sure data is accurate<\/strong><\/li>\n<\/ul>\n

    It is important to make sure the data you capture is accurate otherwise it will be a waste of time and resources you put to capture data.<\/p>\n

      \n
    • We can trust decision made from data<\/strong><\/li>\n<\/ul>\n

      The second reason for doing a debug is to make sure we can easily make the decisions based on data and the data needs to be accurate for this.<\/p>\n

      This could be one of the time-consuming processes, because on the implementation level you might be using an existing template or something that automates this step.<\/p>\n

      Debug, Where?<\/strong><\/h2>\n

      Where can you debug the data? There are multiple stages through which the data passes to reach the final tool. Below are stages through which the data passes:<\/p>\n

        \n
      • Source data<\/strong><\/li>\n<\/ul>\n

        There will always be a source from where the data is coming through. Generally, the data is saved within databases but the data we see is in browsers. We do not work with databases to pull the data so the source data for us will be in the browser. You can see the source code by right-clicking on the page and click on \u2018view source code\u2019 option.<\/p>\n

        Source data is generally coming from a backend server which is written in a programming language like PHP or python.<\/p>\n

          \n
        • Code used to extract data or send it to platform<\/strong><\/li>\n<\/ul>\n

          This would be the front-end code and is generally written in a language like JavaScript. This is generally a pages DOM and its also where most pixels and analytics tools live they consume data from the page and send it to the platform.<\/p>\n

            \n
          • Intermediary data management platform(GTM, Segment)<\/strong><\/li>\n<\/ul>\n

            The intermediary data management tools have made our lives easier by eliminating the above two steps. Now we import the data directly to Google Tag Manager<\/a> or Segment<\/a>\u00a0and use built-in functions and variables to send the data to the destination tool.<\/p>\n

              \n
            • Final tool consuming data<\/strong><\/li>\n<\/ul>\n

              This is the tool where we want to see the data. The data we see in this tool should be accurate and trustworthy so we can make decisions.<\/p>\n

              Before starting to read the actual process, we recommend that you install the Google Analytics & GTM chrome extensions<\/a>. Few the essential extensions you will need for debugging are:<\/p>\n

                \n
              1. Google Tag Assistant<\/a><\/li>\n
              2. Google Analytics Debugger<\/a><\/li>\n
              3. DataLayer Inspector by Analytics Pros<\/a><\/li>\n<\/ol>\n

                Data Pipeline<\/strong><\/h2>\n

                Next, you have to look at the data pipeline. The place where the data exists and how it transmits to the analytics tool. For a website possible points included in the data pipeline would be:<\/p>\n

                  \n
                1. Server<\/li>\n
                2. Browser<\/li>\n
                3. Tag management solution(optional)<\/li>\n
                4. Analytics<\/li>\n<\/ol>\n

                  <\/p>\n

                  Generally, when debugging we do not work with anything on the server-side, so the possible ways to debug data would be in the browser or tag management if used.<\/p>\n

                  <\/a><\/p>\n

                  When we work with the above pipeline, the best practice is to start from the initial step. Usually, when we start debugging, we only check the point where we notice the problem which means we have to verify each individual step that leads up to the point.<\/p>\n

                  So the best practice is to start from the initial step and verify data at each level so we will know how data is passed on to next step.<\/p>\n

                  Server Side Data<\/strong><\/h2>\n

                  We said that we will take this process from the initial point. So our first step is to look for data on server side. Below are few things we can verify at server side:<\/p>\n

                    \n
                  1. Data being rendered from the backend<\/li>\n
                  2. Generally, the data is already available by the time page is loaded<\/li>\n
                  3. We can inspect the page dom to see if it is available<\/li>\n
                  4. In most cases, developers implement this and we only see what is rendered<\/li>\n
                  5. Common issues that occur due to server-side data\n
                      \n
                    1. Formatting, showing up as string<\/li>\n
                    2. Have unnecessary formats(unescaped special character, currency code etc)<\/li>\n
                    3. Time when the data becomes available (on page load, dom ready etc) \u00a0vs when the tool is expecting data<\/li>\n<\/ol>\n<\/li>\n
                    4. Best way to debug:\n
                        \n
                      1. View-source<\/li>\n
                      2. Console.log<\/li>\n
                      3. alert();<\/li>\n<\/ol>\n<\/li>\n<\/ol>\n

                        When to use?<\/strong><\/h3>\n

                        Server-side data is checked when the data is not available after the page is being loaded or the data you see is not in the proper format you expect it to be.<\/p>\n

                        Browser \/ Mobile App<\/strong><\/h2>\n

                        Browser is the most common place to debug and likely where most errors occur. A lot of variety on how data may behave based on browser or device factor. Browser is in the middle of server-side data and everything that comes after it.<\/p>\n

                        Most of the errors that occur are browser based. There are multiple types of browsers including mobile and desktop version. So testing on multiple browsers is a little time-consuming.<\/p>\n

                        The common issues that arise on the browser are:<\/p>\n

                          \n
                        1. Browser compatibility with code<\/li>\n
                        2. Most of the errors exist in your code<\/li>\n<\/ol>\n

                          Best ways to debug<\/p>\n

                            \n
                          1. View-source<\/li>\n
                          2. Console<\/li>\n
                          3. Tool-specific extensions<\/li>\n
                          4. Network request (shows everything that happens in the bowser)<\/li>\n
                          5. Debugger<\/li>\n<\/ol>\n

                            View-Source<\/strong><\/strong><\/h3>\n

                            This is the place where you can see the actual code. You can see the data source to view the actual code placement. For instance, whenever you have any confusions about the tracking code placement, right click on the website and select view-source.<\/p>\n

                            Press Ctrl\/Command+ F and search for \u2018UA-\u2019, to see where the analytics tracking code is placed on the website. Besides this, you can also look for ecommerce tracking and datalayer codes using the view-source method.<\/p>\n

                            Console<\/strong><\/h3>\n

                            Console is the best tool you can use to test your JS & jQuery codes. Before adding the codes in GTM or Analytics, use console mode to test.<\/p>\n

                            <\/p>\n

                            The console is also used for debugging with GA debugger and the datalayer inspector plugin, which are the essential tools for your debugging process.<\/p>\n

                            When to use?<\/strong><\/h3>\n

                            You should use this when there are chances of code errors and issues with programming logic. You will be able to see the actual code behind the loaded page.<\/p>\n

                            Tool Specific Extensions<\/strong><\/h3>\n

                            There are multiple browser extensions available which make it easy to display data in the console. Below are some of the extensions that can help you with the debugging process.<\/p>\n

                            Ability to see what data is being sent<\/p>\n

                              \n
                            1. Google Analytics Debugger<\/li>\n
                            2. Google Tag Assistant<\/li>\n
                            3. Mixpanel Debugger<\/li>\n
                            4. Facebook Pixel Helper<\/li>\n<\/ol>\n

                              These tool specific extensions are good for use because they have organized everything according to best practices and are easy to use.<\/p>\n

                              However, these extensions generally check for best practises so if your site is custom built it may not correctly identify if the setup is correct. Debugging is done when there is something that these extensions don\u2019t pick up or if you still suspect errors despite the extensions giving you an all clear.<\/p>\n

                              When to use?<\/strong><\/h3>\n

                              Tool-specific extensions are used for verifying correct tracking code installation and verifying the best practices.<\/p>\n

                              Network Requests<\/strong><\/h3>\n

                              This is where you can see all the information that is being sent and received. It is known as the communication layer for all information being sent to a browser.<\/p>\n

                              The cons for network request is that when the page renders, It gets data from other sources as well and not only analytics. It is necessary that you apply filters and you should know which filters to apply.<\/p>\n

                              Whenever you are extracting something in network request, you should already know how to extract that particular data. Also to understand the tool, you should know what each parameter means.<\/p>\n

                              For example, in the network tab if you search for google-analytics it would show requests that are being sent to that end-point and all parameters sent along with it.<\/p>\n

                              <\/p>\n

                              You can not always expect to see everything here, because in some case cases some data may be hashed and not easily readable.<\/p>\n

                              When to use?<\/strong><\/h3>\n

                              Network request gives you all the data that is being sent and received so mostly useful for really diving deep to look at data in depth.<\/p>\n

                              Debugger<\/strong><\/h3>\n

                              The main power of debugger is that you can stop the code at the execution time and inspect to see which function is running and what data is passing at the moment.<\/p>\n

                              <\/p>\n

                              The benefit of using debugger is you can see the data being changed while it is executing. You can add breakpoints to stop the code at a certain point and see what data is actually passing at the moment.<\/p>\n

                              When to use?<\/strong><\/h3>\n

                              Using debugger is best for folks with technical background and understanding of javascript.<\/p>\n

                              PS: These are the sort of problems where a lot of folks may reach out to us for support<\/a>.<\/strong><\/p>\n

                              Tag Management Platform<\/strong><\/h2>\n

                              There are multiple tools counted in data management platform which are Google Tag Manager, Tealium, and Segment. There are actually the tools where you can observe and test your data.<\/p>\n

                              For example, preview mode in tag manager helps in testing if your implementation is correct and the right data is passing or not. Similarly, you can use the debugger in Segment for testing purposes.<\/p>\n

                              When to use?<\/strong><\/h3>\n

                              This is optional and should be only used when you are using any of the tag management tool.<\/p>\n

                              Analytics Tool<\/strong><\/h2>\n

                              This is the final place where all the data is being recorded. The data is generally processed before it is displayed in the analytics tool. If you find any data issue at this point, you will have to go back to previous steps, that is why it is recommended to start debugging from the beginning.<\/p>\n

                              This is generally good for seeing patterns on narrowing cause of a certain error. For instance, we can use dimensions that allows us to segment data. Also, we can cross-check data with other platforms like ecommerce orders, sign-ups..etc.<\/p>\n

                              In most cases, there isn\u2019t any actual cause of errors unless you didn\u2019t configure the account correctly. The common configuration errors include time zones, filters, account linking, referral exclusion,..etc.<\/p>\n
